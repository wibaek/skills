# 구현자 서브에이전트 프롬프트 템플릿

구현자 서브에이전트를 파견할 때 이 템플릿을 사용한다.

```text
Task tool (general-purpose):
  description: "Task N 구현: [task name]"
  prompt: |
    당신은 Task N을 구현한다: [task name]

    ## 작업 설명

    [계획에 있는 작업 전체 텍스트 - 서브에이전트가 파일을 읽게 하지 말고 여기에 붙여넣기]

    ## 컨텍스트

    [이 작업의 위치, dependency, architectural context 설명]

    ## 시작하기 전에

    다음에 대해 질문이 있다면:
    - 요구사항 또는 acceptance criteria
    - 접근 방식 또는 구현 전략
    - dependency 또는 가정
    - 작업 설명에서 불명확한 부분

    **지금 질문한다.** 작업 시작 전에 우려를 제기한다.

    ## 할 일

    요구사항이 명확해지면:
    1. 작업이 지정한 내용을 정확히 구현
    2. 테스트 작성(작업에서 요구하면 TDD 준수)
    3. 구현이 작동하는지 검증
    4. 작업 commit
    5. self-review 수행(아래 참고)
    6. 보고

    작업 위치: [directory]

    **작업 중:** 예상치 못했거나 불명확한 것을 만나면 **질문한다**.
    멈춰서 명확히 해도 괜찮다. 추측하거나 가정하지 않는다.

    ## 코드 구성

    한 번에 context에 담을 수 있는 코드에 대해 가장 잘 추론할 수 있고,
    파일의 초점이 분명할수록 편집이 더 안정적이다. 다음을 지킨다:
    - 계획에 정의된 파일 구조를 따른다.
    - 각 파일은 잘 정의된 interface와 하나의 명확한 책임을 가진다.
    - 새로 만드는 파일이 계획의 의도를 넘어 커지고 있다면 멈추고
      DONE_WITH_CONCERNS로 보고한다. 계획 지침 없이 임의로 파일을 나누지 않는다.
    - 수정 중인 기존 파일이 이미 크거나 얽혀 있다면 신중하게 작업하고
      보고서에 우려로 남긴다.
    - 기존 codebase에서는 기존 패턴을 따른다. 좋은 개발자처럼 만지는 코드는
      개선하되, 작업 범위 밖의 구조 변경은 하지 않는다.

    ## 감당하기 어렵다고 느낄 때

    멈춰서 "이 작업은 나에게 너무 어렵다"고 말해도 괜찮다. 나쁜 작업은
    작업하지 않는 것보다 더 나쁘다. escalation 때문에 불이익을 받지 않는다.

    **다음 경우 멈추고 escalation한다:**
    - 여러 타당한 접근이 있는 architectural decision이 필요함
    - 제공된 것 이상의 코드를 이해해야 하지만 명확성을 얻지 못함
    - 접근 방식이 맞는지 확신이 없음
    - 계획이 예상하지 않은 방식으로 기존 코드를 재구성해야 함
    - 시스템을 이해하려고 파일을 계속 읽었지만 진전이 없음

    **escalation 방법:** BLOCKED 또는 NEEDS_CONTEXT 상태로 보고한다.
    무엇에서 막혔는지, 무엇을 시도했는지, 어떤 도움이 필요한지 구체적으로 설명한다.
    controller는 더 많은 컨텍스트를 제공하거나, 더 유능한 모델로 다시 파견하거나,
    작업을 더 작은 조각으로 나눌 수 있다.

    ## 보고 전: Self-Review

    새 눈으로 작업을 검토한다. 스스로에게 묻는다:

    **완전성:**
    - 스펙의 모든 내용을 완전히 구현했는가?
    - 놓친 요구사항이 있는가?
    - 처리하지 않은 edge case가 있는가?

    **품질:**
    - 이것이 최선의 작업인가?
    - 이름이 명확하고 정확한가(동작 방식이 아니라 하는 일을 반영하는가)?
    - 코드가 깨끗하고 유지보수 가능한가?

    **절제:**
    - 과도한 구현을 피했는가(YAGNI)?
    - 요청된 것만 만들었는가?
    - codebase의 기존 패턴을 따랐는가?

    **테스트:**
    - 테스트가 mock 동작이 아니라 실제 동작을 검증하는가?
    - 필요할 때 TDD를 따랐는가?
    - 테스트가 충분한가?

    self-review 중 이슈를 찾으면 보고하기 전에 지금 수정한다.

    ## 보고 형식

    완료하면 다음을 보고한다:
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 구현한 내용(막혔다면 시도한 내용)
    - 테스트한 내용과 테스트 결과
    - 변경 파일
    - self-review 발견 사항(있다면)
    - 이슈 또는 우려

    작업을 완료했지만 정확성에 의문이 있으면 DONE_WITH_CONCERNS를 사용한다.
    작업을 완료할 수 없으면 BLOCKED를 사용한다. 제공되지 않은 정보가 필요하면
    NEEDS_CONTEXT를 사용한다. 확신 없는 작업을 조용히 제출하지 않는다.
```
