# 계획 작성

## 개요

우리 코드베이스에 대한 컨텍스트가 전혀 없고 취향도 의심스러운 엔지니어가 실행한다고 가정하고 포괄적인 구현 계획을 작성한다. 각 작업에서 건드릴 파일, 코드, 테스트, 확인해야 할 문서, 테스트 방법까지 필요한 모든 것을 문서화한다. 계획 전체를 작게 나뉜 작업으로 제공한다. DRY. YAGNI. TDD. 잦은 커밋.

상대는 숙련된 개발자지만, 우리의 도구와 문제 도메인은 거의 모른다고 가정한다. 좋은 테스트 설계도 잘 모를 수 있다고 가정한다.

**시작할 때 알림:** "구현 계획을 만들기 위해 writing-plans 스킬을 사용합니다."

**컨텍스트:** 격리된 worktree에서 작업한다면 실행 시점에 `superpowers:using-git-worktrees` 스킬로 만들어져 있어야 한다.

**계획 저장 위치:** `docs/plans/YYYY-MM-DD-<feature-name>.md`

- 계획 위치에 대한 사용자 선호가 있으면 이 기본값보다 우선한다.

## 범위 확인

스펙이 여러 독립 하위 시스템을 포함한다면 brainstorming 단계에서 하위 프로젝트 스펙으로 나뉘었어야 한다. 그렇지 않다면 하위 시스템별 계획으로 나누자고 제안한다. 각 계획은 독립적으로 작동하고 테스트 가능한 소프트웨어를 만들어야 한다.

## 파일 구조

작업을 정의하기 전에 어떤 파일을 만들거나 수정할지, 각 파일이 무엇을 책임지는지 먼저 매핑한다. 여기서 분해 결정이 확정된다.

- 경계가 명확하고 인터페이스가 잘 정의된 단위로 설계한다. 각 파일은 하나의 명확한 책임을 가져야 한다.
- 한 번에 컨텍스트에 담을 수 있는 코드에 대해 가장 잘 추론할 수 있고, 파일이 집중되어 있을수록 편집도 신뢰할 수 있다. 너무 많은 일을 하는 큰 파일보다 작고 집중된 파일을 선호한다.
- 함께 바뀌는 파일은 함께 둔다. 기술 계층이 아니라 책임 기준으로 나눈다.
- 기존 코드베이스에서는 확립된 패턴을 따른다. 코드베이스가 큰 파일을 사용한다고 해서 일방적으로 구조를 바꾸지는 않는다. 하지만 수정할 파일이 이미 다루기 어려울 정도로 커졌다면 계획에 분리를 포함하는 것은 합리적이다.

이 구조가 작업 분해를 안내한다. 각 작업은 독립적으로 의미 있는 자기완결적 변경을 만들어야 한다.

## 작은 작업 단위

**각 단계는 하나의 행동이다(2-5분):**

- "실패하는 테스트 작성" - 단계
- "실패하는지 실행해 확인" - 단계
- "테스트를 통과시키는 최소 구현" - 단계
- "테스트 실행 후 통과 확인" - 단계
- "커밋" - 단계

## 계획 문서 헤더

**모든 계획은 반드시 이 헤더로 시작한다:**

```markdown
# [Feature Name] 구현 계획

> **agentic worker용:** 필수 하위 스킬: 이 계획을 작업 단위로 구현하려면 `superpowers:subagent-driven-development`(권장) 또는 `superpowers:executing-plans`를 사용한다. 단계 추적에는 checkbox(`- [ ]`) 문법을 사용한다.

**목표:** [무엇을 만드는지 한 문장으로 설명]

**아키텍처:** [접근 방식을 2-3문장으로 설명]

**기술 스택:** [핵심 기술/라이브러리]

---
```

## 작업 구조

````markdown
### Task N: [Component Name]

**파일:**

- 생성: `exact/path/to/file.py`
- 수정: `exact/path/to/existing.py:123-145`
- 테스트: `tests/exact/path/to/test.py`

- [ ] **Step 1: 실패하는 테스트 작성**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **Step 2: 테스트를 실행해 실패 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상: "function not defined"로 FAIL

- [ ] **Step 3: 최소 구현 작성**

```python
def function(input):
    return expected
```

- [ ] **Step 4: 테스트를 실행해 통과 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상: PASS

- [ ] **Step 5: Commit**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 플레이스홀더 금지

모든 단계는 엔지니어에게 실제로 필요한 내용을 포함해야 한다. 아래는 **계획 실패**다. 절대 쓰지 않는다:

- "TBD", "TODO", "나중에 구현", "세부사항 채우기"
- "적절한 error handling 추가" / "validation 추가" / "edge case 처리"
- "위 내용에 대한 테스트 작성"(실제 테스트 코드 없음)
- "Similar to Task N"(코드를 반복한다. 엔지니어가 작업을 순서대로 읽지 않을 수 있다.)
- 무엇을 해야 하는지만 말하고 어떻게 할지 보여주지 않는 단계(코드 단계에는 코드 블록 필수)
- 어떤 작업에서도 정의되지 않은 타입, 함수, 메서드 참조

## 기억할 것

- 항상 정확한 파일 경로
- 모든 단계에 완전한 코드. 단계가 코드를 바꾸면 그 코드를 보여준다.
- 예상 출력이 포함된 정확한 명령
- DRY, YAGNI, TDD, 잦은 커밋

## 자기 검토

전체 계획을 작성한 뒤 스펙을 새 눈으로 보고 계획과 대조한다. 이것은 직접 실행하는 체크리스트이며, 서브에이전트를 파견하는 것이 아니다.

**1. 스펙 커버리지:** 스펙의 각 섹션/요구사항을 훑는다. 그것을 구현하는 작업을 가리킬 수 있는가? 빠진 것을 나열한다.

**2. 플레이스홀더 스캔:** 위 "플레이스홀더 금지" 섹션의 위험 패턴을 계획에서 찾는다. 고친다.

**3. 타입 일관성:** 나중 작업에서 사용한 타입, 메서드 시그니처, 속성 이름이 이전 작업에서 정의한 것과 일치하는가? Task 3에서는 `clearLayers()`였는데 Task 7에서 `clearFullLayers()`라면 버그다.

문제가 있으면 계획 안에서 바로 수정한다. 다시 리뷰할 필요 없이 고치고 넘어간다. 스펙 요구사항에 대응하는 작업이 없다면 작업을 추가한다.

## 실행 인계

계획을 저장한 뒤 실행 방식을 제안한다:

**"`docs/plans/<filename>.md`에 계획을 저장했습니다. 실행 방식은 두 가지입니다:**

**1. Subagent-Driven(권장)** - 작업마다 새 서브에이전트를 파견하고 작업 사이 리뷰를 수행해 빠르게 반복

**2. Inline Execution** - `executing-plans`를 사용해 현재 세션에서 작업을 실행하고 checkpoint 기반으로 batch 실행

**어떤 방식으로 진행할까요?"**

**Subagent-Driven을 선택하면:**

- **필수 하위 스킬:** `superpowers:subagent-driven-development` 사용
- 작업마다 새 서브에이전트 + 2단계 리뷰

**Inline Execution을 선택하면:**

- **필수 하위 스킬:** `superpowers:executing-plans` 사용
- 리뷰 체크포인트가 있는 배치 실행
