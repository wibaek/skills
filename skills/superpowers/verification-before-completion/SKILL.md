# 완료 전 검증

## 개요

검증 없이 작업이 완료되었다고 주장하는 것은 효율이 아니라 부정직이다.

**핵심 원칙:** 언제나 주장보다 증거가 먼저다.

**이 규칙의 글자를 어기는 것은 정신을 어기는 것이다.**

## 철칙

```text
새 검증 증거 없이는 완료 주장을 하지 않는다
```

이 메시지에서 검증 명령을 실행하지 않았다면 통과한다고 주장할 수 없다.

## 게이트 함수

```text
어떤 상태를 주장하거나 만족감을 표현하기 전에:

1. 식별: 이 주장을 증명하는 명령은 무엇인가?
2. 실행: 전체 명령을 실행한다(새로, 완전하게).
3. 읽기: 전체 출력을 읽고, exit code를 확인하고, 실패 수를 센다.
4. 검증: 출력이 주장을 확인하는가?
   - 아니면: 실제 상태를 증거와 함께 말한다.
   - 맞으면: 주장을 증거와 함께 말한다.
5. 그때만: 주장을 한다.

어떤 단계든 건너뛰면 검증이 아니라 거짓말이다
```

## 흔한 실패

| 주장 | 필요 증거 | 충분하지 않은 것 |
| --- | --- | --- |
| 테스트 통과 | 테스트 명령 출력: 실패 0개 | 이전 실행, "통과할 것 같음" |
| 린터 깨끗함 | 린터 출력: 오류 0개 | 부분 확인, 외삽 |
| 빌드 성공 | 빌드 명령: exit 0 | 린터 통과, 로그가 좋아 보임 |
| 버그 수정 | 원래 증상 테스트: 통과 | 코드 변경, 수정됐다고 가정 |
| 회귀 테스트 작동 | red-green cycle 검증 | 테스트가 한 번 통과 |
| 에이전트 완료 | VCS diff가 변경을 보여줌 | 에이전트가 "성공"이라고 보고 |
| 요구사항 충족 | 줄 단위 체크리스트 | 테스트 통과 |

## 위험 신호 - 멈춤

- "should", "probably", "seems to" 사용
- 검증 전에 만족감 표현("Great!", "Perfect!", "Done!" 등)
- 검증 없이 commit/push/PR 하려 함
- 에이전트 성공 보고를 신뢰함
- 부분 검증에 의존함
- "이번 한 번만"이라고 생각함
- 피곤해서 끝내고 싶음
- **실행한 검증 없이 성공을 암시하는 어떤 표현이든 사용**

## 합리화 방지

| 핑계 | 현실 |
| --- | --- |
| "이제 작동할 것 같아" | 검증을 실행한다. |
| "확신해" | 확신은 증거가 아니다. |
| "이번 한 번만" | 예외는 없다. |
| "린터가 통과했어" | 린터는 컴파일러가 아니다. |
| "에이전트가 성공이라고 했어" | 독립적으로 검증한다. |
| "피곤해" | 피로는 핑계가 아니다. |
| "부분 확인이면 충분해" | 부분 확인은 아무것도 증명하지 않는다. |
| "말이 다르니 규칙이 적용되지 않아" | 형식보다 정신이 우선이다. |

## 핵심 패턴

**테스트:**

```text
✅ [테스트 명령 실행] [확인: 34/34 pass] "All tests pass"
❌ "Should pass now" / "Looks correct"
```

**회귀 테스트(TDD Red-Green):**

```text
✅ 작성 -> 실행(통과) -> 수정 되돌림 -> 실행(반드시 실패) -> 복구 -> 실행(통과)
❌ "회귀 테스트를 작성했습니다" (red-green 검증 없음)
```

**빌드:**

```text
✅ [빌드 실행] [확인: exit 0] "Build passes"
❌ "린터가 통과했습니다" (린터는 컴파일을 확인하지 않음)
```

**요구사항:**

```text
✅ 계획 재독 -> 체크리스트 생성 -> 각 항목 검증 -> 누락 또는 완료 보고
❌ "테스트가 통과하니 phase 완료"
```

**에이전트 위임:**

```text
✅ 에이전트 성공 보고 -> VCS diff 확인 -> 변경 검증 -> 실제 상태 보고
❌ 에이전트 보고 신뢰
```

## 왜 중요한가

24개의 실패 기억에서:

- 사람 파트너가 "I don't believe you"라고 말했다. 신뢰가 깨졌다.
- 정의되지 않은 함수가 배포됐다. 실행 시 crash가 났을 것이다.
- 빠진 요구사항이 배포됐다. 기능이 불완전했다.
- 거짓 완료 -> 방향 수정 -> 재작업으로 시간이 낭비됐다.
- 위반: "정직은 핵심 가치다. 거짓말하면 교체된다."

## 적용할 때

**항상 다음보다 먼저 적용한다:**

- 성공/완료 주장의 어떤 변형이든
- 만족감 표현
- 작업 상태에 대한 긍정적 진술
- 커밋, PR 생성, 작업 완료
- 다음 작업으로 이동
- 에이전트에게 위임

**규칙이 적용되는 것:**

- 정확한 문구
- 의역과 동의어
- 성공의 암시
- 완료/정확성을 제안하는 어떤 커뮤니케이션이든

## 결론

**검증에는 지름길이 없다.**

명령을 실행한다. 출력을 읽는다. 그 다음 결과를 주장한다.

협상 불가능하다.
